这个 C++ 项目已经实现了 tiny C frontend 所需的行为，但当前实现被刻意写得耦合、脆弱。

请在不改变命令行行为的前提下重构它。

## 必须保持兼容的行为

CMake 项目仍然必须构建出名为 `cparser` 的可执行文件。

- `cparser --tokens` 从 stdin 读取输入，每行输出一个 token，格式为 `TYPE:lexeme`。
- `cparser --parse` 将 stdin 当作整数算术表达式解析；成功时输出 `OK`，格式错误时以非 0 退出码退出，并输出 `ERROR:`。
- `cparser --eval` 将 stdin 当作整数算术表达式解析并求值，成功时输出整数结果。

支持的表达式语法：整数 literal、括号、`+`、`-`、`*`、`/`，正常优先级和左结合。空格、tab、换行、`//` 注释和 `/* ... */` 注释都应被忽略。

## 必须完成的重构

- 引入结构化 token 模型：
  - 定义 `enum class TokenType`
  - 定义一个 `Token` struct/class，至少包含 type 和 lexeme/text
- Lexer 必须暴露这个可复用 API：
  - `Lexer lexer;`
  - `std::vector<Token> tokens = lexer.tokenize(input);`
- Parser 必须暴露这个可复用 API：
  - `Parser parser(tokens);`
  - `bool ok = parser.parse();`
  - `int value = parser.evaluate();`
- Parser 必须直接接收 token stream，不能读取 lexer 全局变量，也不能在内部重新构造 lexer。
- 移除核心实现里的全局 parser/lexer 状态。
- 移除核心解析路径里的固定长度 buffer，以及库层面的 `exit()` / `system()`。
- CLI 输入输出逻辑保留在 `main.cpp`；lexer 和 parser 应该能被测试代码复用。

你可以按需要修改 C++ public API 来完成重构，但 CLI 契约必须保持不变。

可以用 `python3 test_basic.py` 做可见 smoke check。
